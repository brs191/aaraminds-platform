# Generated Code Handling

A static analyzer that parses source text sees only the source — and on a modern Java codebase, a significant part of the running program is not in the source. This reference covers the annotation-processor and code-generation gap, and how to extract a model that is not silently incomplete.

## The gap — the source is not the program

Java annotation processors and build plugins generate code at compile time. The common ones on a Spring Boot codebase:

- **Lombok** — `@Getter`, `@Setter`, `@Builder`, `@Data`, `@RequiredArgsConstructor` generate getters, setters, builders, constructors, `equals` / `hashCode`.
- **MapStruct** — generates the implementation class behind a mapper interface.
- **Spring** — generates proxy classes around beans for transactions, AOP, and `@Configuration`.
- Others — Immutables, Dagger, generated gRPC / protobuf or OpenAPI clients.

None of those members exist in the `.java` source files. An extractor that parses source text only will not see a single Lombok getter, will see a MapStruct mapper interface with no implementation, and will miss every call edge that runs through any of them. The model looks complete and is quietly missing a layer of the program.

## Why it is not a minor omission

The generated members are load-bearing. A builder is how objects are constructed; a MapStruct mapper is where data is transformed between layers — a data flow; a Spring proxy is where transaction boundaries live. Miss them and the call graph has holes exactly at the construction, mapping, and transaction-boundary points a comprehension model most needs to be right about. Worse, the omission is *silent* — nothing in a source-only parse signals that members are missing.

## The two strategies

Decide between them before the parser is written:

1. **Extract post-annotation-processing.** Run the build's annotation processors and analyze the generated sources or the compiled bytecode. This is the accurate option — the model sees what the compiler saw, every generated member included. The cost: extraction now needs a working build, a heavier prerequisite than parsing loose files. For most comprehension work this is the right choice, and it aligns with the requirement (`ast-extraction-and-parsing.md`) that resolution runs against a buildable checkout anyway.
2. **Model the generated members explicitly.** Encode the known behaviour of each processor — a `@Getter` field implies a `getX()` method, a `@Builder` implies a builder class — and synthesize those members into the model without running the processor. Lighter-weight and build-free, but it is a per-processor effort, it tracks processor behaviour that can shift between versions, and it covers nothing for a processor you have not modeled.

Post-processing extraction is the default; explicit modeling is the fallback when running the build is genuinely not viable.

## Mark synthesized members as such

However the generated members enter the model, tag them with their origin — generated by Lombok, by MapStruct, a Spring proxy. A reviewer reading the model should be able to tell a hand-written method from a generated one, and an evidence link to "the `@Getter` annotation that produced this" is more honest than presenting a synthesized getter as authored source.

## Worked example — the Code Intelligence Factory

The CIF schema flags this directly as a v0.1 open question and "a real Java-specific risk for M1": Lombok, MapStruct, and other annotation processors generate methods not in the source AST, and extraction must either run against post-annotation-processing output or model the generated members explicitly. It is called out as a risk rather than left to be discovered — which is the right move: the generated-code strategy is a decision made before the M1 graph builder is written, because retrofitting it means re-running every extraction.

## Verification questions

1. Has the codebase been inventoried for annotation processors and code generators — Lombok, MapStruct, Spring proxies, others?
2. Is there an explicit strategy — post-processing extraction, or explicit modeling — chosen before the parser was built?
3. If post-processing: does extraction run against a built checkout so the processors have run?
4. If explicit modeling: is each processor's behaviour modeled, and is the coverage gap for unmodeled processors known and stated?
5. Are generated members tagged with their origin, distinguishable from hand-written code?
6. Does the call graph include edges through generated members — builders, mappers, proxies?
